- **Real-world**: Uses actual SONiC OS, not lightweight SAIVS emulator

Why Avoid Path B On Windows
- Path B1/B2 require SAIVS containers with veth network interfaces
- Veth setup is problematic on Windows/WSL2 and causes dataplane failures
- FdbAttributeTest and other dataplane tests will hang/fail
- These paths work better on Linux native environments only

---------------------------------------
Goal of Phase 1
- Put the lab into a known, clean state before starting tests.
- Remove hidden leftovers from older sessions (containers, VM processes).

Why this matters
- Old containers can keep ports busy.
- Old VM processes can create confusing behavior.
- A clean baseline makes failures easier to debug.

What each Phase 1 command does
1. Stop QEMU process if it exists.
- Prevents old SONiC VM instances from interfering.

2. Stop all running Docker containers.
- Ensures no old runtime is still active.

3. Remove known lab containers.
- Clears named containers from previous attempts.

4. List all containers after cleanup.
- Confirms what is still present.

	Important note before Phase 4
	-----------------------------
	Phase 4 usually feels slower than Phase 3.

	Reason:
	- Phase 3 is a simple API sanity test.
	- Phase 4 uses legacy SAI-PTF style execution, which does more setup.
	- If pytest must be downgraded inside the container, package installation adds extra wait time.

	Best practice:
	- Run Phase 4 as separate commands, not one long chained command.
	- That makes progress visible and avoids the feeling that the terminal is stuck.
