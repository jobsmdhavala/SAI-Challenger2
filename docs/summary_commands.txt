Phase 3 - Path A: QEMU SONiC DUT + Docker Client (✅ WINDOWS RECOMMENDED)
=========================================================================
This is the officially recommended setup for Windows.
Reference: windows_sonic_vs_repro_setup.md

TERMINAL 1: Start SONiC VM (keep running in background)
-------------------------------------------------------
& "C:\Program Files\qemu\qemu-system-x86_64.exe" `
  -m 6144 -smp 4 `
  -drive file="C:\Users\mdhavala\Downloads\sonic-vs.qcow2",if=virtio,format=qcow2 `
  -netdev user,id=mgmt,hostfwd=tcp:127.0.0.1:2222-:22,hostfwd=tcp:127.0.0.1:6379-:6379 `
  -device e1000,netdev=mgmt `
  -nographic

Wait 2-3 minutes for SONiC to boot. You'll see login prompt when ready.


TERMINAL 2: Start Docker client + SSH forwarder
------------------------------------------------
$repo = (Get-Location).Path

# Step 3.1 - Start SAI-Challenger client container
docker run -d --name sc-client-run `
  --cap-add=NET_ADMIN `
  --add-host host.docker.internal:host-gateway `
  -v "${repo}:/sai-challenger" `
  plvisiondevs/sc-client:bookworm-latest

# Step 3.2 - Start SSH forwarder (forwards port 22 -> VM port 2222)
docker run -d --name sonic-ssh-fwd `
  -p 127.0.0.1:22:22 `
  alpine/socat `
  tcp-listen:22,reuseaddr,fork `
  tcp:host.docker.internal:2222

# Step 3.3 - Verify containers started
docker ps --filter name="sc-client-run|sonic-ssh-fwd" --format "table {{.Names}}\t{{.Status}}"

Expected: Both containers should show "Up X seconds"

# Step 3.4 - Test connectivity from client to SONiC VM
docker exec -i sc-client-run bash -lc "apt-get update >/dev/null 2>&1; apt-get install -y netcat-openbsd >/dev/null 2>&1 || true; nc -vz host.docker.internal 22; nc -vz host.docker.internal 6379"

Expected: Two successful connections (port 22 and 6379 reachable)

# Step 3.5 - Run first test: VLAN creation (API-only)
docker exec -i sc-client-run bash -lc "cd /sai-challenger && pytest -v tests/api/test_vlan.py -k test_vlan_create --testbed=sainpu_sonic_vs"

Expected result: ✅ PASSED

# Step 3.6 - Run dataplane test: L2 access-to-access forwarding
docker exec -i sc-client-run bash -lc "cd /sai-challenger && pytest tests/test_l2_basic.py::test_l2_access_to_access_vlan --testbed=sainpu_sonic_vs -v"

TROUBLESHOOTING QUICK REFERENCE
================================

If Phase 3 tests FAIL:

1️⃣ SONiC VM not booting?
   - Verify QEMU window shows startup messages
   - Wait 2-3 minutes - syncd/database need time to initialize
   - Check QEMU process running: Get-Process qemu-system-x86_64

2️⃣ Docker containers not starting?
   - Check logs: docker logs sc-client-run and docker logs sonic-ssh-fwd
   - Verify Docker daemon running: docker ps
   - Try restart: docker restart sc-client-run sonic-ssh-fwd

3️⃣ Connectivity tests fail (nc -vz)?
   - Verify SONiC VM is fully booted (wait another minute)
   - Check QEMU forwarding: netstat -an | grep 6379
   - Verify SSH forwarder: docker logs sonic-ssh-fwd

4️⃣ Pytest tests hang or timeout?
   - This usually means veth dataplane issues (Path B)
   - You should be using Phase 3 (Path A)
   - Check testbed file: ensure using sainpu_sonic_vs.json (not saivs_*)

5️⃣ FdbAttributeTest fails/hangs?
   - Expected to fail on Path B (veth issues)
   - Use Phase 3 (Path A) instead - FdbAttributeTest works there
   - Ensure using --testbed=sainpu_sonic_vs

6️⃣ "Connection refused" for Redis/SSH?
   - Network isolation issue
   - Restart containers: docker restart sonic-ssh-fwd sc-client-run
   - Restart SONiC VM and wait 2-3 minutes
   - Check host firewall isn't blocking loopback ports
