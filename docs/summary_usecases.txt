SAI-Challenger Virtual Use Cases (Ubuntu WSL, No Hardware)
==========================================================

Purpose
-------
- List every practical virtual use case you can run on your laptop.
- Keep only software-only paths.
- Map each use case to status from current validation.

Core Mental Model
-----------------
Every run is a combination of:
1) DUT type (virtual only in this guide)
2) SAI API backend (Thrift or Redis)
3) Traffic style (API-only or software dataplane)
4) Runtime mode (standalone or client-server)


Virtual Use Cases You Asked For
-------------------------------
Use Case 1 - SAIVS API-only
- DUT: SAIVS virtual switch
- Backend: Redis/Thrift abstraction through standalone SAIVS runtime
- Traffic: None (object CRUD only)
- Mode: Standalone
- Why use it: Fast confidence check for API behavior
- Current status: PASS (validated)
- Evidence: test_vrf.py -> 2 passed

Use Case 2 - SAIVS software dataplane
- DUT: SAIVS virtual switch
- Backend: SAI runtime in container
- Traffic: Software packet dataplane (L2 scenarios)
- Mode: Standalone
- Why use it: Best no-hardware learning path for VLAN/LAG/FDB behavior
- Current status: PASS (validated)
- Evidence: test_l2_basic.py -> 12 passed

Use Case 3 - SONiC virtual DUT control-plane
- DUT: SONiC VM in QEMU
- Backend: Redis (sainpu_sonic_vs testbed)
- Traffic: API-only
- Mode: Client-server style (client container + SONiC VM)
- Why use it: Learn SONiC-specific Redis behavior in virtual setup
- Current status: PASS (validated)
- Evidence: test_vlan_create/test_vlan_remove -> 2 passed

Use Case 4 - SONiC virtual DUT dataplane
- DUT: SONiC VM in QEMU
- Backend: Redis
- Traffic: Software dataplane tests from client side
- Mode: Client-server style
- Why use it: Try full SONiC dataplane behavior without hardware
- Current status: EXECUTED, PROFILE-LIMITED
- Evidence: test_l2_access_to_access_vlan failed due to missing/insufficient bridge-port list (IndexError)
- Interpretation: this VM/profile currently lacks dataplane port assumptions required by the test


What Is Not Included (Hardware Required)
----------------------------------------
- Physical switch DUT validation
- Hardware traffic-generator workflows
- Fanout-switch topologies with physical links
- Any scenario requiring real front-panel cables/devices


Recommended Order (Ubuntu WSL)
------------------------------
1) Use Case 1 (SAIVS API-only)
2) Use Case 2 (SAIVS dataplane)
3) Use Case 3 (SONiC QEMU API)
4) Use Case 4 (SONiC QEMU dataplane, only if profile supports enough ports)

Why this order
- Fast wins first
- Maximum learning without hardware
- Avoid time loss on profile-dependent SONiC dataplane constraints


Simple Comparison
-----------------
SAIVS path
- Pros: fastest, stable, full software dataplane learning
- Best for: daily no-hardware learning and repeatable labs

SONiC QEMU path
- Pros: closest to SONiC Redis architecture
- Best for: control-plane and SONiC behavior understanding
- Caveat: dataplane depends on image/profile port availability


Final Answer To Your Main Question
----------------------------------
Can all virtual/no-hardware use cases be done on Ubuntu WSL?
- Yes for SAIVS API and SAIVS dataplane (validated).
- Yes for SONiC QEMU API (validated).
- SONiC QEMU dataplane is possible in principle, but your current image/profile is port-limited for this test set.

Practical rule:
- If your goal is full virtual learning now, use SAIVS for dataplane and SONiC QEMU for Redis control-plane.
