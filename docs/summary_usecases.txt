SAI-Challenger Use Cases - Simple Learning Guide
================================================

Purpose
- This file simplifies the main repo use-cases and architecture for your laptop lab.
- It focuses only on what is practical without physical hardware.


1) Simple Mental Model
----------------------
SAI-Challenger becomes easy when you break it into 4 choices:

1. What is the DUT?
- Virtual DUT
- Physical DUT

2. How do we configure the DUT?
- Redis client
- Thrift client

3. How do we send traffic?
- No traffic, API-only
- PTF/Scapy software traffic
- OTG/snappi traffic generator

4. How is the runtime arranged?
- Standalone mode
- Client-server mode

If you remember only one thing, remember this:
- SAI-Challenger test logic stays mostly the same.
- The testbed JSON chooses the backend details.

-------------------

3) What the Main Repo README Is Really Saying
---------------------------------------------
The README lists many permutations because SAI-Challenger is flexible.

In plain English:
- You can test a virtual switch or a real switch.
- You can configure it through Redis or Thrift.
- You can test only API CRUD or include packet traffic.
- You can run everything in one container or split client/server.

That is why the project looks big.
The core idea is still simple:
- same test framework
- different testbed choices


4) Applicable Use Cases For Windows Laptops (RECOMMENDED ORDER)
----------------------------------------------------------

⭐ USE CASE A FIRST: Virtual SONiC DUT via QEMU (Windows-Proven)
- **RECOMMENDED STARTING POINT for Windows**
- Uses QEMU VM (SONiC OS) + Docker client
- Port forwarding: SSH (2222) + Redis (6379)
- Real SONiC behavior, not emulation
- Reference: windows_sonic_vs_repro_setup.md
- Example: Path A: QEMU SONiC VM + sainpu_sonic_vs testbed
- Tests work reliably: test_vlan_create, test_l2_basic, etc.

⚠️  USE CASE B ADVANCED: Virtual DUT with Local SAIVS (Linux-Only)
- **NOT RECOMMENDED for Windows** - has fundamental issues:
  - Requires veth network interfaces (problematic on Windows/WSL2)
  - Dataplane tests hang or fail (FdbAttributeTest, L2 tests)
  - Container networking between client/server is fragile on Windows
- Use these only on Linux native or WSL2 with special setup
- Path B1: SAIVS Thrift (limited, no dataplane)
- Path B2: SAIVS with legacy PTF (dataplane issues)

⌛ FUTURE: Once Path A is solid, Path B may be useful for:
- Understanding standalone SAIVS architecture
- Linux-specific development workflows
- Advanced testing scenarios (on proper Linux)

Three use-case details follow (for reference only):

4A) Applicable Use Cases For Ubuntu/Linux (RECOMMENDED ORDER)
-------------------------------------------------------------

USE CASE A FIRST: SAIVS Thrift API (B1)
- Fastest way to validate SAI API CRUD basics.
- Single runtime container in standalone mode.
- Good first confidence check on Linux.

USE CASE B SECOND: SAI-PTF Umbrella (B2)
- Best way to learn legacy SAI-PTF integration.
- Dataplane/veth behavior is usually stable on Ubuntu/Linux.
- Good path for FdbAttributeTest and L2 packet scenarios.

USE CASE C THIRD: SONiC DUT via QEMU (Path A)
- Use when you want SONiC-specific behavior and Redis workflow.
- Closer to production SONiC architecture.

Recommended order for Ubuntu/Linux:
1. Path B1 first
2. Path B2 second
3. Path A third

7) Architecture Docs - What They Mean For You
----------------------------------------------
The architecture docs are useful, but they are easier when translated into simple language.

Architecture overview
- SAI-Challenger sits between the test and the DUT.
- Test code uses a unified format.
- Client adapter converts that format into Redis or Thrift calls.

Simple picture:
```mermaid
flowchart LR
    A[Test Case] --> B[Unified SAI commands]
    B --> C{SAI Client Adapter}
    C --> C1[Redis Client]
    C --> C2[Thrift Client]
    C1 --> D[DUT]
    C2 --> D[DUT]
```

Why this matters:
- Your test code does not need to change much when backend changes.
- The testbed file decides whether Redis or Thrift is used.


8) SAI Client Docs - What They Mean For You
-------------------------------------------

9) Running Tests Docs - What They Mean For You
----------------------------------------------
Composing testbed definition
- This is always relevant.
- The testbed JSON is where you choose:
  - ASIC
  - target
  - SKU
  - client type
  - dataplane type
  - ports

Running in standalone mode
- Very relevant for you.
- Simplest path for API-first learning.
- Best current practical starting point.

Running in client-server mode
- Relevant later.
- Better when you want clearer separation of client and DUT runtime.
- More useful for traffic-based experiments.
- On Windows host, this is usually more fragile than API-first flow.

Using SONiC device as SAI DUT
- Relevant to your QEMU SONiC VM path.
- Important when you want Redis + SONiC behavior.
- Not the easiest first step, but very useful after Path B is stable.


10) Testbed Definition - Simple Meaning
---------------------------------------
Think of the testbed JSON as the lab wiring diagram.

It answers these questions:
- What DUT am I testing?
- Which API backend am I using?
- What dataplane am I using?
- Which ports/interfaces are mapped?

Simple view:
```mermaid
flowchart TD
    A[Testbed JSON] --> B[NPU section]
    A --> C[Dataplane section]
    B --> B1[asic]
    B --> B2[target]
    B --> B3[sku]
    B --> B4[client type and config]
    C --> C1[type]
    C --> C2[mode]
    C --> C3[port_groups]
```
